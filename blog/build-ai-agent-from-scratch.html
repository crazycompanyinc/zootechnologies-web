How to Build an AI Agent From Scratch

Complete 2026 guide: architecture, memory, tool execution, orchestration, and production deployment.

May 9, 2026 18 min read ZOO Technologies

The AI agent space is exploding. Every week brings new frameworks, new demos, and new promises. But most tutorials stop at the "Hello World" stage — a single LLM call with a prompt. Real agents are different. They have memory, use tools, make decisions, handle errors, and operate autonomously.

This guide walks you through building a production-grade AI agent from scratch. No frameworks. No abstractions hiding the complexity. Just the core patterns that actually matter.

What Makes an Agent an Agent?

Before writing code, let's define what we're building. An AI agent is a system that:

  • Perceives its environment (user input, tool results, external data)
  • Reasons about what to do next (using an LLM as the decision engine)
  • Acts by executing tools, calling APIs, or modifying state
  • Remembers past interactions and learned context
  • Adapts its behavior based on feedback and outcomes

This is fundamentally different from a chatbot. A chatbot responds. An agent decides and acts.

Architecture: The Core Loop

Every agent runs on a variation of the same loop:

while not task_complete:
    context = build_context(memory, history, tools)
    decision = llm.reason(context)
    if decision.type == "tool_call":
        result = execute_tool(decision.tool, decision.args)
        memory.store(result)
    elif decision.type == "respond":
        return decision.response
    elif decision.type == "plan":
        subtasks = decompose(decision.goal)
        for subtask in subtasks:
            execute(subtask)

This seems simple. The complexity is in the details: context management, error handling, tool orchestration, and state persistence.

Step 1: The Reasoning Engine

The LLM is your agent's brain. But raw LLM calls aren't enough. You need structured output — the agent must decide what to do, not just what to say.

const systemPrompt = `You are an autonomous agent. For each step, output JSON:
{
  "thought": "your reasoning",
  "action": "tool_name or 'respond'",
  "args": { "param": "value" }
}

Available tools: search, read_file, write_file, execute_code`;

const response = await llm.chat({
  messages: [...history, { role: "user", content: task }],
  response_format: { type: "json_object" }
});

const decision = JSON.parse(response.content);

Key insight: always use structured output. Free-text responses make it impossible to reliably parse agent decisions. JSON mode or function calling is essential.

Step 2: Memory Systems

Without memory, your agent is amnesiac. Every interaction starts from zero. There are three types of memory you need:

Short-Term Memory (Context Window)

The conversation history. Limited by the LLM's context window. Manage it aggressively — summarize old messages, drop irrelevant turns, prioritize recent context.

Long-Term Memory (Vector Store)

Persistent knowledge the agent accumulates. Store embeddings of past interactions, documents, and facts. Retrieve relevant context using semantic search.

// Store a memory
await vectorStore.add({
  text: "User prefers Python over TypeScript for agent development",
  metadata: { type: "preference", timestamp: Date.now() }
});

// Retrieve relevant memories
const memories = await vectorStore.search(
  "What language should I use for this agent?",
  { limit: 5, threshold: 0.7 }
);

Working Memory (State)

The agent's current task state: what it's doing, what it's learned, what's next. This is typically a structured object that gets updated each loop iteration.

Step 3: Tool Execution

Tools are how agents interact with the world. Every tool needs:

  • A clear description (the LLM uses this to decide when to use it)
  • Typed inputs (JSON schema for arguments)
  • Error handling (what happens when the tool fails)
  • Output formatting (structured results the agent can reason about)
const tools = {
  search: {
    description: "Search the web for information. Use when you need current facts.",
    parameters: {
      type: "object",
      properties: {
        query: { type: "string", description: "Search query" },
        max_results: { type: "number", default: 5 }
      },
      required: ["query"]
    },
    execute: async ({ query, max_results }) => {
      try {
        const results = await webSearch(query, max_results);
        return { success: true, data: results };
      } catch (error) {
        return { success: false, error: error.message };
      }
    }
  }
};

Critical pattern: Always return structured results with success/failure status. The agent needs to know if a tool failed and why.

Step 4: Error Handling & Recovery

Production agents fail. A lot. Network timeouts, API rate limits, malformed outputs, infinite loops. Your agent needs to handle all of these:

  • Retry with backoff for transient failures (network, rate limits)
  • Circuit breakers for consistently failing tools
  • Max iteration limits to prevent infinite loops
  • Graceful degradation — if a tool fails, try an alternative approach
  • Human escalation — when the agent can't resolve something, ask for help
const MAX_ITERATIONS = 20;
let iterations = 0;

while (!task_complete && iterations < MAX_ITERATIONS) {
  iterations++;
  try {
    const decision = await agent.reason(context);
    const result = await execute(decision);
    context.update(result);
  } catch (error) {
    if (isRetryable(error)) {
      await sleep(Math.pow(2, retryCount) * 1000);
      continue;
    }
    return { error: "Task failed", details: error.message };
  }
}

Step 5: Multi-Agent Orchestration

Complex tasks require multiple agents working together. The most common patterns:

  • Manager/Worker: A manager agent decomposes tasks and delegates to specialized workers
  • Pipeline: Agents process data in sequence, each adding value
  • Peer-to-Peer: Agents collaborate as equals, sharing state through a common store
// Manager pattern
const manager = new Agent({
  role: "Decompose tasks and delegate to specialists",
  tools: [delegate_to_researcher, delegate_to_coder, delegate_to_writer]
});

const researcher = new Agent({ role: "Research and gather information", tools: [search, read_file] });
const coder = new Agent({ role: "Write and review code", tools: [write_file, execute_code] });
const writer = new Agent({ role: "Write reports and documentation", tools: [write_file] });



Multi-agent systems are where the real power lies — and where most of the complexity hides. Start with a single agent. Add complexity only when the task demands it.



Step 6: Production Deployment



Getting an agent to work on your laptop is step one. Running it in production requires:



    

  • Observability: Log every decision, tool call, and outcome. You need to debug agent behavior in production.
    • 

  • Cost management: LLM calls are expensive. Cache results, use smaller models for simple decisions, set spending limits.
    • 

  • Security: Agents with tool access are powerful — and dangerous. Sandbox execution, limit tool scope, validate all inputs.
    • 

  • Versioning: Version your prompts, tools, and agent configurations. You need to roll back when things break.
    • 




Common Mistakes (And How to Avoid Them)



    

  1. Too many tools, too fast. Start with 3-5 tools. Add more only when the agent consistently needs them.
    2. 

  2. Ignoring token costs. Every context token costs money. Summarize aggressively. Use the smallest model that works.
    3. 

  3. No human-in-the-loop. For high-stakes decisions, always have a human approval step.
    4. 

  4. Over-engineering. Don't build a multi-agent system for a task a single agent can handle.
    5. 

  5. Skipping observability. If you can't see what your agent is doing, you can't improve it.
    6. 




The Bottom Line



Building AI agents is not about the LLM. It's about the system around the LLM — memory, tools, error handling, orchestration, and observability. The LLM is the engine. Everything else is the car.



Start simple. Build a single agent with 3 tools. Get it working end-to-end. Then add complexity incrementally. The teams that ship agents fastest are the ones that resist the urge to over-engineer from day one.
